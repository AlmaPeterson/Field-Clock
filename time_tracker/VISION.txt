# FieldClock — App Vision & Development Plan
> A photo-first time tracking app built for construction workers and tradespeople.

---

## The Problem

Construction workers lose money and credibility because time tracking is painful.
Handwritten logs get lost. Memory is unreliable. Managers can't verify hours.
There's no easy way to prove what you did, when you did it, and how long it took.

## The Solution

FieldClock turns your phone camera into a time clock and work journal.
Every task starts and ends with a photo. The app does the math.
At the end of the day, you send your manager a clean, verified summary —
with timestamps, locations, photos, and total hours. No paperwork. No disputes.

---

## Core Principles

- **Offline-first** — Works on job sites with no signal. Photos stay local; cloud backup syncs when connected.
- **Photo-verified** — Every time entry is backed by a real image with a timestamp.
- **One-handed simplicity** — Designed for workers with dirty gloves. Big buttons, minimal taps.
- **Manager-ready output** — Summaries are formatted for approval, not just personal reference.
- **No data loss** — Time records are too important to lose. Cloud backup offloads photos; local is the cache.

---

## Who Uses This App

### Primary User — The Worker (You)
- Construction worker, tradesperson, or contractor
- Works on multiple tasks per day across one or more job sites
- Needs to prove hours worked to a manager or client
- Often has dirty hands, poor lighting, and no signal on site
- Values speed and simplicity over features

### Secondary User — The Manager
- Receives daily summaries via share (WhatsApp, email, SMS, etc.)
- Needs to approve or dispute hours
- Wants photo evidence alongside time records
- Does not need to install the app

## Feature Plan

### Phase 1 — MVP ✅ Done

| Feature | Status | Description |
|---|---|---|
| Clock In / Clock Out | ✅ | Photo taken at start and end of workday. Timestamp locked automatically. |
| Task Tracking | ✅ | Take a "before" photo to start a task, "after" photo to end it. |
| Auto Time Calculation | ✅ | Time per task calculated automatically, rounded to nearest 15 minutes. |
| Task Naming | ✅ | Name each task after completing it (e.g. "Framing — Master Bedroom"). |
| Time Editing | ✅ | Manually adjust start/end time if you forgot to log something. |
| Daily Summary Screen | ✅ | Clean overview: clock in/out, list of tasks, total hours for the day. |

## Daily Workflow (How It Works)

```
Arrive on site
    → Open app → Clock In → Camera takes photo → Day starts

Start a task
    → Tap "New Task" → Camera takes BEFORE photo → Timer starts

Finish a task
    → Tap "End Task" → Camera takes AFTER photo → Timer stops
    → Name the task → App rounds to nearest 15 min → Task saved

Repeat for each task throughout the day...

End of day
    → Tap "Clock Out" → Camera takes photo → Day ends
    → Review Daily Summary → Edit any times if needed
    → Tap Share → Choose: Full Day (with photos) or No Photos
    → Send to manager via WhatsApp / Email / SMS
```

## Tech Stack

| Layer | Choice | Reason |
|---|---|---|
| Framework | Flutter | Cross-platform, single codebase, great camera support |
| Language | Dart | Flutter native |
| Local Database | SQLite (via `sqflite`) | Offline-first, reliable, fast |
| Camera | `camera` package | Already integrated |
| Location | `geolocator` + `geocoding` | GPS tagging per task |
| File Storage | `path_provider` | Local photo storage |
| PDF Generation | `pdf` package | Export and invoice |
| Sharing | `share_plus` | Native share sheet |
| Notifications | `flutter_local_notifications` | Clock-out reminders |
| State Management | Provider (ChangeNotifier) | Simple and maintainable |

## Data Models (Actual Schema — DB version 4)

```
Worker
  - id, name, hourly_rate, created_at

Job
  - id, name, address, client_name, start_date, end_date, status

WorkDay
  - id, job_id, date
  (clock in/out state comes from Sessions, not stored directly)

Session  ← one clock-in/clock-out pair; multiple allowed per day
  - id, work_day_id, clock_in_time, clock_out_time

Task  ← container: name, category, notes, location
  - id, work_day_id, name, division, notes, location
  (no time fields — time lives on TaskSession)

TaskSession  ← one start/end pair; multiple allowed per task (resume support)
  - id, task_id, start_time, end_time

TaskPhoto  ← before / after / general; multiple allowed per task
  - id, task_id, type, file_path, timestamp
```

**Key rules:**
- Task has no `startTime`, `endTime`, `duration`, or photo fields — those all live on TaskSession / TaskPhoto
- WorkDay has no `clockInTime` / `clockOutTime` fields — derived from Session records
- Duration rounding always goes through `TimeUtils.roundToNearest15()`
- Multi-job per day is not yet supported (WorkDay has one `job_id`); planned for Phase 2

## Known Issues / Next Up

- **Edit previous days is broken** — the edit_day_screen flow has bugs; fixing this is the next priority
- **Multi-job per day** — current schema limits one job per WorkDay; needs a model change before implementing
- **Photo storage growth** — no cleanup strategy yet; cloud backup (Phase 3) is the long-term answer

---

## Success Metrics

- Worker can clock in, complete 5 tasks, and send a summary in under 3 minutes total overhead per day
- Manager can read and approve a summary without needing to ask follow-up questions
- Zero data loss — records survive app reinstall or phone replacement
- Works fully offline on a job site with no cell signal
